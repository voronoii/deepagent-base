# 추가 미들웨어 적용 가이드

전세 계약/보증 상담 챗봇 기준으로, 현재 구성에 추가하면 효과적인 미들웨어를 정리한다.

---

## 현재 구성 현황

| 미들웨어/기능 | 적용 위치 | 상태 |
|-------------|----------|------|
| MemoryMiddleware (AGENTS.md) | orchestrator | ✅ |
| FilesystemMiddleware (backend) | orchestrator | ✅ |
| SkillsMiddleware (hug_rag.md) | orchestrator | ✅ |
| SubAgentMiddleware (task) | orchestrator | ✅ |
| SummarizationMiddleware | 자동 적용 | ✅ |
| PatchToolCallsMiddleware | 자동 적용 | ✅ |
| TracingMiddleware | 모든 서브에이전트 | ✅ |
| LawVerificationMiddleware | risk-assessment-agent | ✅ |
| RetryPolicy | LangGraph 노드 | ✅ |
| MemorySaver checkpointer | LangGraph | ✅ |

---

## 추가 권장 미들웨어

### 1. ToolCallLimitMiddleware — 우선순위: 높음

**대상**: orchestrator + 모든 서브에이전트

**이유**: vLLM 기준 32k context window에서 서브에이전트가 tool call을 반복하면 context가 빠르게 소진된다. 특히 `risk-assessment-agent`는 `LawVerificationMiddleware`가 모델을 재호출하므로, 법령 참조 → 검증 요청 → 또 법령 참조 → 또 검증 같은 루프가 이론적으로 가능하다. 현재 `_VERIFICATION_MARKER`로 1회 방어하고 있지만, 다른 도구 호출 루프는 방어가 없다.

**적용 예시**:

```python
# orchestrator
inner_agent = create_deep_agent(
    ...
    middleware=[ToolCallLimitMiddleware(max_calls=15)],
)

# 서브에이전트는 더 타이트하게
risk_assessment_agent_spec = {
    ...
    "middleware": [
        TracingMiddleware("risk-assessment-agent"),
        LawVerificationMiddleware(),
        ToolCallLimitMiddleware(max_calls=8),
    ],
}
```

### 2. RiskResponseStructureMiddleware — 우선순위: 높음

**대상**: risk-assessment-agent

**이유**: `LawVerificationMiddleware`는 "RAG 도구를 사용했는가"를 검증하지만, 응답 내용의 구조는 검증하지 않는다. 전세 상담에서 위험도 분석 결과가 일관된 포맷(위험도 등급 + 법적 근거 + 권고사항)으로 나와야 오케스트레이터가 제대로 취합하고, 프론트엔드에서도 파싱이 가능하다.

**적용 예시**:

```python
class RiskResponseStructureMiddleware(AgentMiddleware):
    """위험도 분석 응답에 필수 항목이 포함되어 있는지 검증."""

    REQUIRED_SECTIONS = ["위험도", "법적 근거", "권고사항"]
    _MARKER = "[STRUCTURE_CHECK]"

    @hook_config(can_jump_to=["model"])
    def after_model(self, state, runtime):
        last = state["messages"][-1]
        text = getattr(last, "content", "") or ""

        # 무한 루프 방지
        for msg in state["messages"]:
            if self._MARKER in (getattr(msg, "content", "") or ""):
                return None

        missing = [s for s in self.REQUIRED_SECTIONS if s not in text]
        if missing:
            return {
                "messages": [HumanMessage(
                    content=(
                        f"응답에 다음 항목이 누락되었습니다: {', '.join(missing)}. "
                        f"반드시 위험도(상/중/하), 법적 근거(조문), 권고사항을 포함하세요.\n"
                        f"{self._MARKER}"
                    )
                )],
                "jump_to": "model",
            }
        return None
```

**효과**: 사용자가 "이 특약 위험한가요?"라고 물었을 때, 항상 위험도 등급 + 법적 근거 + 권고사항이 포함된 일관된 응답이 나온다.

### 3. SummarizationToolMiddleware — 우선순위: 중간

**대상**: orchestrator

**이유**: 자동 `SummarizationMiddleware`(85% 초과 시 작동)와 별개로, 에이전트가 스스로 판단하여 압축을 트리거할 수 있는 `compact_conversation` 도구를 제공한다. 전세 상담은 한 세션에서 여러 조항을 순차적으로 분석하는 경우가 많다("1조 분석해줘 → 2조도 → 특약도"). 이런 패턴에서 32k window는 빠르게 차는데, 85% 자동 트리거를 기다리기보다 에이전트가 중간중간 알아서 정리하면 더 효율적이다.

**적용**: `create_deep_agent`에서 `SummarizationToolMiddleware` 활성화 방식 확인 필요 (옵션 플래그 또는 middleware 리스트에 직접 추가).

---

### 4. ContractContextMiddleware — 우선순위: 중간

**대상**: orchestrator

**이유**: 사용자가 이전에 입력한 계약 정보(보증금 액수, 계약 기간, 물건 주소 등)를 세션 중에 기억하고, 후속 질문에 자동으로 맥락을 붙여주면 UX가 향상된다. 예를 들어 사용자가 처음에 "보증금 2억, 서울 마포구 아파트"라고 했으면, 이후 "이 특약 위험한가요?"라는 질문에 해당 맥락이 자동으로 포함된다.

**적용 예시**:

```python
class ContractContextMiddleware(AgentMiddleware):
    """이전 대화에서 추출된 계약 정보를 모델 호출 전에 주입."""

    def before_model(self, state, runtime):
        context = self._load_contract_context(state)
        if context:
            return {
                "messages": [SystemMessage(
                    content=f"[사용자 계약 정보]\n{context}"
                )]
            }
        return None

    def _load_contract_context(self, state):

        ...
```

**비고**: `FilesystemBackend` 또는 LangGraph `Store`를 활용하여 구현할 수 있으며, 구현 복잡도에 따라 미들웨어가 아닌 다른 방식이 나을 수 있다.

## 추가하지 않아도 되는 것

| 후보 | 불필요 이유 |
|------|-----------|
| 입력 분류 미들웨어 | 오케스트레이터 프롬프트의 라우팅 규칙이 충분히 명확함 ("전세계약 → risk-agent, 그 외 → research-agent") |
| 가드레일 미들웨어 | 프롬프트의 "risk-assessment-agent는 반드시 전세계약에 관련된 질문만 처리" 규칙으로 충분 |
| TodoListMiddleware 커스텀 | 이미 자동 제공됨. 전세 상담은 단일 질의-응답 패턴이 주이므로 멀티스텝 계획이 자주 필요하지 않음 |

## 적용 우선순위 요약

| 순위 | 미들웨어 | 대상 | 핵심 이유 |
|------|---------|------|----------|
| **1** | ToolCallLimitMiddleware | orchestrator + 모든 서브에이전트 | 32k context 보호, 무한 루프 방지 |
| **2** | RiskResponseStructureMiddleware | risk-assessment-agent | 분석 결과 일관성 보장, 프론트엔드 파싱 |
| **3** | SummarizationToolMiddleware | orchestrator | 멀티 조항 분석 시 context 효율화 |
| **4** | ContractContextMiddleware | orchestrator | 세션 내 계약 정보 자동 맥락 유지 |

1, 2번은 안정성/품질에 직결되므로 우선 적용하고, 3, 4번은 실제 사용 패턴을 확인한 후 추가한다.
